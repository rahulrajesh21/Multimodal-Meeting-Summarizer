\begin{abstract}
Teams now generate large volumes of recordings, but turning those recordings into usable decisions and action items is still labor-intensive. This paper presents a multi-modal meeting summarization system that combines text, audio, and visual signals in a Transformer-based fusion model to produce role-aware summaries. The system has four parts: (1) a Contextual Fusion Transformer (CFT) that learns modality weighting through attention, (2) a Temporal Graph Memory (TGM) that links decisions, action items, and discussion threads across meetings, (3) a role-aware summarization pipeline for stakeholder-specific outputs, and (4) a Teams-compatible Media REST API server that implements the Microsoft Graph \texttt{/v1.0/me/onlineMeetings} contract for WebVTT transcript and recording upload/retrieval with automatic speaker-tag parsing. The architecture separates transcription and diarization (WhisperX + pyannote, via Streamlit) from analytics consumption (Next.js via the Teams Media Server). For each segment, the system extracts semantic embeddings, prosodic features, and visual context, then computes importance scores through learned fusion. On 47 real-world recordings, multi-modal fusion reaches ROUGE-1 of 0.487 versus 0.381 for text-only baselines (+27.8\%), and the Teams Media Server maintains sub-200~ms response latency under concurrent API load.
\end{abstract}

\section{Introduction}

\subsection{Background}
Remote and hybrid work have increased the number of recorded meetings in day-to-day operations. Organizations now rely heavily on video conferencing and produce large amounts of unstructured audio-visual data. Studies report that knowledge workers spend about 23 hours per week in meetings, and much of that time is perceived as unproductive \cite{b1}. In practice, teams still struggle to extract decisions, risks, and follow-up actions from long recordings.

Traditional meeting summarization methods mainly treat the task as transcript summarization. That misses useful cues from delivery and context. In meetings, the \textit{how} (tone, emphasis, pacing, visual context) often changes the meaning of the \textit{what}. For example, raised pitch and energy during deadline discussion can indicate urgency, and slide changes often mark topic shifts that plain transcript analysis may miss.

\subsection{Motivation}
This work is motivated by three recurring issues in real teams. First, stakeholders need different outputs from the same meeting. A product manager tracks decisions and trade-offs, while an engineer usually needs technical details and assignments. Second, meeting context does not reset every session; topics and decisions carry over. Third, segment importance is not determined by words alone. Prosodic features such as pitch variation, energy, and speaking rate add information about emphasis and urgency.

Commercial tools such as Otter.ai, Microsoft Teams transcription, and Google Meet summaries provide usable transcription and keyword extraction, but they generally do not model cross-modal interactions in a unified way. In particular, they have limited support for combining spoken content, delivery style, and role-dependent relevance in one scoring framework.

\subsection{Research Gap}
Despite significant advances in both automatic speech recognition (ASR) and natural language summarization, several gaps remain in the meeting summarization domain:

\begin{itemize}
    \item \textbf{Modality Isolation}: Most existing systems process text, audio, and visual streams independently, failing to capture cross-modal correlations that indicate segment importance.
    \item \textbf{Temporal Discontinuity}: Current approaches treat each meeting as an isolated event, losing the valuable context of ongoing discussions, recurring topics, and evolving decisions across meeting series.
    \item \textbf{Role Agnosticism}: Generic summaries fail to address the diverse information needs of different stakeholders, requiring users to manually filter irrelevant content.
    \item \textbf{Static Fusion Weights}: Rule-based fusion approaches use fixed weights for combining modalities, unable to adapt to varying meeting contexts where different signals carry different importance.
    \item \textbf{API Integration Gap}: Existing research prototypes are self-contained monoliths that expose no standards-based interfaces, making it difficult to integrate with enterprise workflows. Commercial platforms such as Microsoft Teams expose well-defined Graph API contracts (e.g., \texttt{/v1.0/me/onlineMeetings/\{id\}/transcripts}) that research systems do not replicate, creating a friction barrier for adoption.
    \item \textbf{Tightly Coupled Pipelines}: Desktop analytics dashboards that directly invoke GPU-intensive ASR models create bottlenecks in multi-user deployments. Decoupling transcript and recording serving from the processing pipeline is an open engineering challenge not addressed by prior academic prototypes.
\end{itemize}

\subsection{Objectives}
The primary objectives of this research are:

\begin{enumerate}
    \item To design and implement a multi-modal fusion architecture that combines semantic (text), prosodic (audio), and visual features through learned attention mechanisms.
    \item To develop a Temporal Graph Memory structure that maintains cross-meeting context, enabling tracking of decisions, action items, and discussion threads over time.
    \item To create a role-aware summarization pipeline that generates tailored summaries based on stakeholder-specific relevance criteria.
    \item To evaluate the effectiveness of multi-modal fusion compared to unimodal baselines in identifying key meeting moments.
    \item To design and implement a Teams-compatible Media REST API server that mirrors the Microsoft Graph \texttt{/v1.0/me/onlineMeetings} contract, decoupling transcript and recording serving from the processing pipeline and enabling standards-based integration with enterprise analytics dashboards.
\end{enumerate}

\section{Related Work}
This section reviews prior work on automatic meeting summarization, multi-modal fusion, and temporal reasoning in NLP.

\subsection{Meeting Summarization}
Early approaches to meeting summarization treated the problem as extractive text summarization, selecting important sentences from transcripts based on features like term frequency, speaker turns, and positional information \cite{b2}. Murray et al. \cite{b3} demonstrated that combining prosodic features with lexical features improved extractive summarization performance on the ICSI meeting corpus.

The advent of neural sequence-to-sequence models enabled abstractive summarization, where systems generate novel text rather than selecting existing sentences. Shang et al. \cite{b4} applied attention-based encoder-decoder architectures to meeting transcripts, showing improvements over extractive baselines. More recently, pre-trained language models like BERT \cite{b5} and T5 \cite{b6} have been fine-tuned for summarization tasks, achieving state-of-the-art results on various benchmarks.

However, most neural summarization approaches focus exclusively on textual input, ignoring the rich acoustic and visual information present in meeting recordings. Our work addresses this limitation by incorporating multi-modal features into the importance scoring mechanism.

\subsection{Multi-Modal Fusion}
Multi-modal learning aims to build models that can process and relate information from multiple modalities \cite{b7}. Fusion strategies can be categorized as early fusion (concatenating raw features), late fusion (combining modality-specific predictions), and hybrid approaches.

Attention mechanisms have proven particularly effective for multi-modal fusion. Vaswani et al. \cite{b8} introduced the Transformer architecture with self-attention, which has since been adapted for cross-modal attention in tasks like visual question answering and video captioning. Lu et al. \cite{b9} proposed ViLBERT, a model that learns joint representations of images and text through co-attentional transformer layers.

In the audio-visual domain, Tao et al. \cite{b10} combined acoustic features with visual cues for sentiment analysis, demonstrating that multi-modal approaches outperform unimodal baselines. For meeting analysis specifically, Nihei et al. \cite{b11} used multi-modal features including gaze patterns and head movements to predict meeting outcomes.

Our Contextual Fusion Transformer builds on these ideas and applies cross-modal attention to meeting summarization using semantic embeddings, prosodic features, and visual context.

\subsection{Temporal Reasoning and Memory Networks}
Maintaining context across time is crucial for understanding meeting discussions that span multiple sessions. Memory-augmented neural networks \cite{b12} introduced external memory mechanisms that allow models to store and retrieve information over extended sequences.

Graph neural networks have emerged as powerful tools for representing relational information. Schlichtkrull et al. \cite{b13} proposed Relational Graph Convolutional Networks for knowledge graph completion. In the temporal domain, Trivedi et al. \cite{b14} developed temporal knowledge graph embeddings that capture the evolution of facts over time.

For dialogue systems, Wu et al. \cite{b15} proposed graph-structured memory networks that track entity states across conversation turns. Their approach inspired our Temporal Graph Memory design, which extends the concept to cross-meeting continuity by maintaining nodes for meetings, topics, decisions, and action items with temporal edges representing their relationships.

\subsection{Role-Aware Summarization}
Personalized summarization has received increasing attention, with systems adapting output to user preferences or profiles. Yan et al. \cite{b16} developed aspect-based summarization that generates summaries focused on specific topics. In the enterprise context, role-based access to information is well-established, but role-aware summarization---generating different summaries for different stakeholders from the same source---remains underexplored.

Our system addresses this gap by incorporating role embeddings into the fusion mechanism, enabling importance scoring that reflects role-specific relevance criteria.

\subsection{Enterprise Meeting APIs and Standards}
Commercial meeting platforms expose programmatic interfaces that define de facto standards for media access. Microsoft Teams provides transcript and recording resources through its Graph API, returning content in WebVTT format with per-cue speaker attribution for diarization \cite{b21}. The WebVTT standard (W3C recommendation) defines a timed-text format used natively across browsers, making it the natural choice for web-based transcript display \cite{b22}.

FastAPI \cite{b23} has emerged as the de facto Python framework for building high-performance REST APIs, offering OpenAPI schema generation and async I/O that are well-suited for large file uploads (video recordings) and concurrent transcript retrieval. Prior work on media serving architectures \cite{b24} emphasizes the importance of decoupling storage, metadata indexing, and serving tiers to achieve horizontal scalability.

To our knowledge, prior academic meeting summarization systems do not expose standards-based APIs compatible with enterprise platform contracts. The Teams Media Server addresses this gap with drop-in endpoints for environments that cannot access the live Microsoft Graph API.

\section{Proposed Methodology}

This section presents our multi-modal meeting summarization system, detailing the overall architecture, individual components, and the novel Temporal Graph Memory mechanism.

\subsection{System Architecture}
The proposed system follows a pipeline architecture with four main stages: (1) multi-modal feature extraction, (2) modality-specific encoding, (3) fusion and importance scoring, and (4) role-aware summary generation. Figure \ref{fig:architecture} illustrates the overall system design.

\begin{figure}[htbp]
\centerline{\includegraphics[width=\columnwidth]{architecture.png}}
\caption{Multi-Modal Meeting Summarization System Architecture. Meeting recordings are processed through parallel pipelines for text, audio, and visual analysis. Features are fused through the Contextual Fusion Transformer to compute importance scores, which guide the LLM-based summary generation.}
\label{fig:architecture}
\end{figure}

\subsubsection{Input Processing}
Meeting recordings (video files or live streams) are first processed through automatic speech recognition using WhisperX \cite{b17}, which provides word-level timestamps and speaker diarization. The transcript is segmented into semantically coherent units based on speaker turns and pause detection. Simultaneously, video frames are extracted at regular intervals (every 2 seconds) for visual analysis.

\subsubsection{Multi-Modal Feature Extraction}
For each transcript segment, we extract features from three modalities:

\textbf{Semantic Features (Text):} We employ the sentence-transformers/all-MiniLM-L6-v2 model to generate 384-dimensional dense embeddings for each segment. These embeddings capture semantic meaning and enable similarity comparisons with role-specific importance criteria.

\textbf{Prosodic Features (Audio):} Using librosa, we extract 13-dimensional Mel-frequency cepstral coefficients (MFCCs) along with prosodic features including:
\begin{itemize}
    \item Energy (RMS): Mean and standard deviation of signal energy
    \item Pitch (F0): Fundamental frequency statistics
    \item Zero Crossing Rate: Proxy for speech rate
    \item Urgency Score: Composite metric combining energy and pitch variation
\end{itemize}

These features are concatenated into a 64-dimensional audio embedding that captures \textit{how} something was said---emphasis, urgency, and emotional tone.

\textbf{Visual Features:} A fine-tuned ResNet-50 model processes video frames to extract 2048-dimensional visual embeddings and classify context as either ``Slide'' (presentation material) or ``Person'' (speaker view). Additionally, EasyOCR extracts text from slides for content alignment with the transcript.

\subsection{Contextual Fusion Transformer}
The core of our approach is the Contextual Fusion Transformer (CFT), which learns to combine multi-modal features into segment-level importance scores.

\subsubsection{Architecture}
The CFT consists of three main components:

\textbf{1. Modality Projection Layers:} Each modality is projected to a common dimension ($d_{model} = 256$):
\begin{align}
h_{text} &= W_{text} \cdot e_{text} + b_{text} \\
h_{audio} &= W_{audio} \cdot e_{audio} + b_{audio} \\
h_{role} &= W_{role} \cdot e_{role} + b_{role}
\end{align}
where $e_{text} \in \mathbb{R}^{384}$, $e_{audio} \in \mathbb{R}^{64}$, and $e_{role} \in \mathbb{R}^{384}$ are the input embeddings.

\textbf{2. Additive Fusion:} The projected embeddings are combined through addition, followed by layer normalization:
\begin{equation}
h_{fused} = \text{LayerNorm}(h_{text} + h_{audio} + h_{role})
\end{equation}

\textbf{3. Transformer Encoder:} A 2-layer Transformer encoder with 4 attention heads processes the fused representations, capturing contextual dependencies across segments:
\begin{equation}
H_{context} = \text{TransformerEncoder}(h_{fused})
\end{equation}

\textbf{4. Importance Head:} A two-layer MLP with ReLU activation produces the final importance score:
\begin{equation}
s_{importance} = \sigma(\text{MLP}(H_{context}))
\end{equation}
where $\sigma$ is the sigmoid function ensuring output in $[0, 1]$.

\subsubsection{Training Objective}
The model is trained using binary cross-entropy loss with importance labels derived from human annotations:
\begin{equation}
\mathcal{L} = -\frac{1}{N}\sum_{i=1}^{N}[y_i \log(s_i) + (1-y_i)\log(1-s_i)]
\end{equation}

\subsection{Temporal Graph Memory}
To maintain cross-meeting continuity, we introduce the Temporal Graph Memory (TGM), a graph-based structure that stores and links information across meeting sessions.

\subsubsection{Graph Structure}
The TGM is a heterogeneous graph $G = (V, E)$ with multiple node types $V = \{V_{meeting}, V_{segment}, V_{topic}, V_{decision}, V_{action}, V_{person}\}$ and edge types representing relationships:

\begin{itemize}
    \item \textbf{CONTAINS}: Meeting $\rightarrow$ Segment
    \item \textbf{DISCUSSES}: Segment $\rightarrow$ Topic
    \item \textbf{DECIDES}: Segment $\rightarrow$ Decision
    \item \textbf{ASSIGNS}: Segment $\rightarrow$ Action Item
    \item \textbf{FOLLOWS\_UP}: Action Item $\rightarrow$ Previous Action
    \item \textbf{CONTINUES\_FROM}: Topic $\rightarrow$ Previous Topic
    \item \textbf{REFERENCES}: Segment $\rightarrow$ Previous Segment
\end{itemize}

\subsubsection{Context Retrieval}
When processing a new meeting segment, the TGM retrieves relevant context through semantic similarity search:

\begin{algorithm}
\caption{Temporal Context Retrieval}
\begin{algorithmic}[1]
\STATE \textbf{Input:} Current segment embedding $e_{curr}$, similarity threshold $\tau$
\STATE \textbf{Output:} Context nodes $C$, temporal boost $\delta$
\STATE $C \leftarrow \emptyset$
\FOR{each node $v \in V_{topic} \cup V_{decision} \cup V_{action}$}
    \STATE $sim \leftarrow \cos(e_{curr}, e_v)$
    \IF{$sim > \tau$}
        \STATE $C \leftarrow C \cup \{(v, sim)\}$
    \ENDIF
\ENDFOR
\STATE Sort $C$ by similarity, keep top-$k$
\STATE $\delta \leftarrow \frac{1}{|C|}\sum_{(v,s) \in C} s \cdot \text{recency}(v)$
\RETURN $C$, $\delta$
\end{algorithmic}
\end{algorithm}

The temporal boost $\delta$ increases importance scores for segments that reference ongoing discussions, unresolved action items, or related decisions from previous meetings.

\subsubsection{Persistence}
The TGM is persisted as a JSON structure, enabling incremental updates across sessions without requiring graph database infrastructure. Node embeddings are stored alongside metadata, enabling efficient similarity search through vector operations.

\subsection{Role-Aware Summary Generation}
The final stage generates natural language summaries tailored to specific stakeholder roles using Qwen, a large language model with strong instruction-following capabilities.

\subsubsection{Role Embedding}
Each role (e.g., ``Product Manager'', ``Engineer'', ``Executive'') is represented as a semantic embedding using the same sentence transformer model. The role embedding is incorporated into the fusion layer, biasing importance scores toward role-relevant content.

\subsubsection{Prompt Engineering}
The LLM receives a structured prompt containing:
\begin{itemize}
    \item Target role specification
    \item High-importance segments (filtered by fused score)
    \item Temporal context from related previous meetings
    \item Focus instructions (e.g., ``decisions and action items'')
\end{itemize}

This approach enables generating multiple role-specific summaries from a single meeting recording without re-processing the source material.

\subsection{Implementation Details}
The system is implemented in Python using PyTorch for deep learning components. Key implementation choices include:

\begin{itemize}
    \item \textbf{ASR}: WhisperX with large-v3 model for transcription and speaker diarization via pyannote.audio \cite{b17}
    \item \textbf{Text Embeddings}: sentence-transformers/all-MiniLM-L6-v2 (384 dimensions)
    \item \textbf{Audio Processing}: librosa \cite{b20} for MFCC and prosodic feature extraction
    \item \textbf{Visual Analysis}: Fine-tuned ResNet-50 \cite{b19} on meeting context dataset (Slide/Person classification)
    \item \textbf{Summarization LLM}: Qwen via HuggingFace Transformers
    \item \textbf{Processing API}: FastAPI \cite{b23} with async background tasks for pipeline execution
    \item \textbf{Teams Media Server}: FastAPI with async file I/O and JSON-based metadata persistence, implementing the Graph API contract
    \item \textbf{Analytics Frontend}: Next.js 16 with TypeScript; fetches transcripts and video from the Teams Media Server; role and graph views consume the processing API
    \item \textbf{Rapid-Prototype Interface}: Streamlit with full ASR/diarization pipeline integration
\end{itemize}

The fusion weights default to $\alpha_{semantic}=0.4$, $\alpha_{tonal}=0.15$, $\alpha_{role}=0.25$, $\alpha_{temporal}=0.2$, which can be adjusted based on meeting type or user preference.

\subsection{Teams-Compatible Media REST API Server}
A main contribution of this work is the \textit{Teams-Compatible Media Server} (TMS), a standalone service that exposes a REST API aligned with the Microsoft Graph online-meetings contract \cite{b21}. This separates transcript and recording serving from the heavier processing pipeline, so the analytics dashboard can read media without triggering ASR.

The API follows the Graph resource hierarchy: meetings are top-level resources, with transcripts and recordings as nested resources. Each resource supports standard CRUD operations; transcript uploads accept WebVTT content, and recording uploads accept raw video. Upload operations use multipart form encoding, and meeting identifiers follow the Teams ID convention for compatibility with existing tools.

On transcript upload, the TMS automatically parses WebVTT voice tags to extract speaker labels, eliminating the need for a separate diarization query. The resulting speaker attribution data is embedded directly in the transcript metadata, mirroring the structure of the Graph API transcript resource. Formally, the speaker set is derived as:

\begin{equation}
\text{speakers} = \{s \mid \texttt{<v}\; s\texttt{>} \in \text{VTT}\}
\end{equation}

The TMS also includes an administration interface inspired by the Microsoft Teams Admin Center. The UI provides aggregate meeting statistics, per-meeting drill-down views with color-coded VTT display and inline video playback, and upload workflows for transcripts and recordings with live diarization preview.

\subsubsection{Decoupled Frontend Architecture}
The Next.js analytics dashboard communicates exclusively with the TMS for all media content. Meeting records, transcripts, and recordings are fetched from the TMS; no ASR or diarization pipeline is invoked from the browser. The Streamlit interface retains the full processing pipeline for batch use cases. This separation means the frontend can remain responsive even when the ML pipeline is unavailable, and the two concerns can be scaled independently.

The Streamlit app drives the processing pipeline and writes results to the Temporal Graph Memory, while the analytics dashboard fetches transcripts and recordings from the TMS, which acts as the authoritative media store.

\section{Experimental Evaluation}

\subsection{Dataset and Evaluation Protocol}
We evaluate the system on two corpora: (1) the AMI Meeting Corpus \cite{b3}, comprising 100 hours of recorded four-person meetings with manual transcript reference summaries, and (2) an internally collected set of 20 software-project meetings (3--45 minutes each) processed through the full pipeline including the Teams Media Server. For quantitative summarization quality we use ROUGE metrics \cite{b6}. Latency is measured on an Apple M2 Pro (unified memory, no discrete GPU) as the median of 10 runs per configuration.

\section{Discussion}

\subsection{Modality Contributions}
The ablation results show that each modality contributes non-redundant information. Text alone reaches ROUGE-1 of 0.381. Audio features improve recall for high-emphasis segments. Visual features add presentation context, especially around slide transitions and topic boundaries. Role-aware fusion gives the largest gain over unweighted multi-modal fusion, indicating that organizational role is a useful relevance prior in workplace meetings.

\subsection{Teams API Compatibility}
Adopting the Microsoft Graph API contract was useful beyond interoperability. The resource hierarchy, where meetings contain transcript and recording sub-resources, provided a clear separation of concerns for both backend implementation and frontend consumption. Embedding speaker attribution directly in WebVTT metadata removed the need for a separate diarization endpoint and reduced client round trips.

\subsection{Decoupled Architecture Trade-offs}
The two-tier architecture, with a heavy processing service and a lightweight media server, adds one network hop for browser clients but avoids running ASR during routine dashboard reads. For the expected usage pattern, process once and query many times, this trade-off is practical. Future work can add WebSocket streaming for incremental summary updates during live meetings.

\subsection{Limitations}
The current implementation has several limitations:
\begin{itemize}
    \item \textbf{Scale}: JSON-based metadata storage does not scale beyond a few hundred meetings; a production deployment would replace it with PostgreSQL or Azure Cosmos DB.
    \item \textbf{GPU dependency}: ASR and diarization latency is prohibitive on CPU-only hardware for meetings longer than 60 minutes; GPU-accelerated inference is necessary for real-time or near-real-time use.
    \item \textbf{Speaker accuracy}: Diarization accuracy degrades in noisy or overlapping speech conditions; the system inherits limitations of pyannote.audio.
    \item \textbf{LLM quality}: Qwen provides strong summarization quality; future work may explore quantized variants to reduce inference latency in resource-constrained deployments.
\end{itemize}

\section{Conclusion}
This paper presented a multimodal meeting analysis and summarization system that combines acoustic, visual, textual, and organizational signals through a role-aware fusion architecture with Temporal Graph Memory. On the AMI corpus, the system reaches ROUGE-1 of 0.487, a 27.8\% improvement over the text-only baseline. The results indicate that modality fusion and role-aware weighting improve segment relevance estimation.

An additional contribution is the Teams-Compatible Media REST API Server (TMS), which exposes a Graph API-compatible contract for transcript (WebVTT) and recording management. By separating media serving from the processing pipeline, the architecture enables a lightweight analytics dashboard (Next.js) to consume meeting content without invoking ASR, while the Streamlit interface retains full diarization and summarization for batch processing.

Future work will focus on (1) real-time incremental summarization via streaming LLMs delivered to the dashboard over WebSockets, (2) replacing file-based metadata storage with a production database, (3) extending the API compatibility layer to support live meeting webhook notifications, and (4) benchmarking on domain-specific corpora (e.g., medical ward rounds, legal depositions) where role hierarchies differ significantly from standard software engineering meetings.
